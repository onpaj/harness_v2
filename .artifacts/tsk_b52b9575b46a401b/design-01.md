# Design: sub-hourly `harness service autoupdate` schedules

No UI: this is a CLI + launchd driver feature. `api/` and `projection.py` are
untouched (invariant #5); there is no board surface for this at all.

## Component design

### 1. `drivers/launchd.py` тАФ new pure builders, existing seam reused

The module already separates *pure builders* (`wrapper_script`, `plist_bytes`)
from the *thin `launchctl` shell* (`load`/`unload`/`status`). The new pieces
slot into the same two buckets тАФ nothing about `load`/`unload`/`status`
changes, since a periodic LaunchAgent is loaded/unloaded/inspected exactly
like the `KeepAlive` one; only the plist content and wrapper differ.

**`parse_interval_minutes(text: str) -> int`**

```python
def parse_interval_minutes(text: str) -> int:
    """Parse "<N>m" / "<N>h" / "<N>d" into seconds. Whole units only."""
```

- Regex `^(\d+)([mhd])$`, case-insensitive on the suffix only (`M`, `H`, `D`
  accepted; the digits are digits).
- Multiplier table: `{"m": 60, "h": 3600, "d": 86400}`.
- `N == 0` тЖТ `ServiceError('--every must be at least 1m, got "0m"')`.
- No match at all (bad suffix, seconds suffix `s`, decimal `1.5m`, bare
  integer `90`, negative `-5m`) тЖТ `ServiceError` with the offending text
  echoed back and the accepted grammar spelled out, e.g.:
  `ServiceError('invalid --every "90s": expected <N>m, <N>h or <N>d "
  "(minutes/hours/days), e.g. "15m"')`.
- Returns `int` seconds; never raises anything but `ServiceError`, so callers
  in `cli.py` only need one `except` clause (mirrors how `load()` already
  raises `ServiceError` for the CLI to catch).

Reuses `ServiceError` rather than introducing a second exception type тАФ it is
already "a service operation failed, with an actionable message," which this
is too, just at parse time before any `launchctl` call happens.

**`periodic_plist_bytes(...) -> bytes`** тАФ a sibling function, not a
generalization of `plist_bytes`

```python
def periodic_plist_bytes(
    *,
    label: str,
    wrapper: Path,
    working_dir: Path,
    log_dir: Path,
    home: Path,
    start_interval_seconds: int,
) -> bytes:
```

Design choice: **a sibling function**, not `plist_bytes(..., start_interval_seconds=None)`.
Rationale тАФ the two plists differ in kind, not by one optional field:
`KeepAlive: True` (supervise forever) vs. `StartInterval` + no `KeepAlive` key
(fire, exit, wait) are mutually exclusive launchd semantics, and a single
function with an `if start_interval_seconds:` branch buried inside would make
both call sites (`_service_install`, `_service_autoupdate_install`) harder to
read than two five-line functions. `plist_bytes` itself is untouched тАФ its
existing tests keep passing unmodified.

```python
definition = {
    "Label": label,
    "ProgramArguments": ["/bin/bash", str(wrapper)],
    "RunAtLoad": True,          # catches up promptly after a reboot/missed window
    "StartInterval": start_interval_seconds,
    "WorkingDirectory": str(working_dir),
    "StandardOutPath": str(log_dir / "harness-autoupdate.log"),
    "StandardErrorPath": str(log_dir / "harness-autoupdate.error.log"),
    "EnvironmentVariables": {"HOME": str(home)},
}
```

No `KeepAlive` key at all (not `False` тАФ launchd treats an *absent* key and
`False` the same for this purpose, but omitting it reads as "this axis does
not apply to a periodic job" rather than "supervision is turned off").

**`autoupdate_wrapper_script(...) -> str`**

```python
def autoupdate_wrapper_script(
    *,
    harness: Path,
    service_label: str,
    path_entries: list[str],
) -> str:
```

```bash
#!/usr/bin/env bash
#
# Generated by `harness service autoupdate install` тАФ regenerated on every install.
# Edit the install command, not this file.

set -euo pipefail

export PATH="{path}"

exec "{harness}" update --restart-service "{service_label}"
```

No `GITHUB_TOKEN` dance here at all (NFR: "simpler than the existing wrapper,
not harder") тАФ `harness update` needs no token, only `PATH` to find `uv`.
This resolves plan Open-Question #1: the wrapper **execs `harness update`**,
it does not re-implement `uv tool upgrade` in bash. It resolves Open-Question
#2 the same way the plan defaulted: `--restart-service` is a new flag on the
*existing* `_update`/`harness update`, so the before/after version comparison
and the kickstart call live once, in Python, unit-testable with
`subprocess`-mocking the way `_update`'s tests already do тАФ not duplicated as
untestable bash. `exec` (not a plain call) so the wrapper process itself
becomes `harness update`, matching the existing wrapper's `exec "{harness}"
run ...` convention and keeping one process, one PID, one exit code for
launchd to observe.

### 2. `cli.py` тАФ `_update` grows a restart flag; three new handlers

**`_update` extended, not duplicated**

```python
update.add_argument(
    "--restart-service",
    default=None,
    dest="restart_service",
    metavar="LABEL",
    help="kickstart the given LaunchAgent label after a version change",
)
```

`_update(args)` gains, after the existing "now: ..." print and *before* the
existing static "restart it with ..." hint:

```python
changed = before_commit_or_version != after_commit_or_version
if args.restart_service:
    if changed:
        _launchctl(["kickstart", "-k", f"gui/{os.getuid()}/{args.restart_service}"])
        print(f"restarted service {args.restart_service} (version changed)")
    else:
        print(f"service {args.restart_service} left running (no version change)")
    return 0
# ... existing manual-mode hint, unchanged, when --restart-service is absent
```

`before`/`after` comparison reuses `installed_commit()` (falls back to
`version_string()`'s version when there is no commit, i.e. a PyPI-style
install) captured once *before* the `uv tool upgrade` subprocess call and once
via `installed_version_report()` *after* тАФ both helpers already exist and are
already called by `_update` today; this only adds a variable to hold the
"before" snapshot. `--restart-service` omitted keeps `_update`'s current
manual-mode output byte-for-byte (regression safety for the existing
`test_update_*` tests) тАФ the new branch is strictly additive, gated on the
flag being present. A failed `uv tool upgrade` still returns non-zero *before*
reaching the restart branch at all, satisfying FR-2 AC4 for free.

**Three new handlers**, same shape as `_service_install/_uninstall/_status`,
placed directly after them:

```python
def _service_autoupdate_install(args: argparse.Namespace) -> int: ...
def _service_autoupdate_uninstall(args: argparse.Namespace) -> int: ...
def _service_autoupdate_status(args: argparse.Namespace) -> int: ...
```

`_service_autoupdate_install`:
1. `_require_macos()` guard тАФ identical shape to `_service_install`.
2. `harness.drivers.launchd.parse_interval_minutes(args.every)`, wrapped in a
   `try/except ServiceError` that prints `f"error: {error}"` to stderr and
   returns `2` (an argument-validation exit code, matching how a bad
   `--every` is a usage error, not a runtime one).
3. `service_entry_point()` тАФ reused verbatim; same existence check as
   `_service_install`.
4. Build `root / "harness-autoupdate.sh"` via `autoupdate_wrapper_script(...)`,
   `chmod(0o755)`.
5. Build `plist_path(home, args.label)` via `periodic_plist_bytes(...)`.
6. `load(os.getuid(), target, args.label)` тАФ the *same* `load()` used by
   `_service_install`; its bootout-then-bootstrap dance already gives FR-1
   AC6 (idempotent re-install) with zero new code.
7. Print label/wrapper/plist/logs/interval, mirroring `_service_install`'s
   final block.

`_service_autoupdate_uninstall` / `_service_autoupdate_status`: thin
delegations to `unload`/`status`/`plist_path` exactly like their non-autoupdate
counterparts тАФ no new logic, only the different default `--label`.
`_service_autoupdate_status` additionally decodes `StartInterval` back to a
human string (`"every 15m"`) by reading the installed plist with `plistlib`
and dividing by the unit table `parse_interval_minutes` used тАФ the inverse
mapping lives next to the parser in `launchd.py` as
`format_interval(seconds: int) -> str` (pick the largest exact unit: `3600`
тЖТ `"1h"`, `5400` тЖТ falls back to `"90m"` since `h` doesn't divide evenly).

**Argument wiring** тАФ a sub-subparser under `service`, mirroring `service`
itself under `subparsers`:

```python
autoupdate = service_actions.add_parser(
    "autoupdate", help="periodically run `harness update` and restart the service"
)
autoupdate_actions = autoupdate.add_subparsers(dest="autoupdate_action", required=True)

autoupdate_install = autoupdate_actions.add_parser("install")
autoupdate_install.add_argument("--root", default=None)
autoupdate_install.add_argument("--every", required=True)
autoupdate_install.add_argument("--label", default=f"{DEFAULT_LABEL}.autoupdate")
autoupdate_install.add_argument("--service-label", default=DEFAULT_LABEL, dest="service_label")
autoupdate_install.set_defaults(handler=_service_autoupdate_install)

autoupdate_uninstall = autoupdate_actions.add_parser("uninstall")
autoupdate_uninstall.add_argument("--label", default=f"{DEFAULT_LABEL}.autoupdate")
autoupdate_uninstall.set_defaults(handler=_service_autoupdate_uninstall)

autoupdate_status = autoupdate_actions.add_parser("status")
autoupdate_status.add_argument("--label", default=f"{DEFAULT_LABEL}.autoupdate")
autoupdate_status.set_defaults(handler=_service_autoupdate_status)
```

`--every` is `required=True` (resolves plan Open-Question #4: no implicit
default schedule тАФ an operator must state a granularity). `service_actions`
itself needs no change beyond the one new `add_parser("autoupdate", ...)`
call; `action="autoupdate"` on the outer `service` parser routes into the
inner `autoupdate_action` exactly the way `service`/`run`/`init` already route
via `handler=` on the leaf parser, so `args.handler(args)` at the bottom of
`main()` needs no new dispatch logic at all.

## Data schemas

No changes to `models.py`, `Task`, queues, or the board projection тАФ this
feature is entirely two on-disk artifacts, following the existing convention:

| Artifact | Path | Built by |
|---|---|---|
| Autoupdate wrapper | `<root>/harness-autoupdate.sh` | `autoupdate_wrapper_script` |
| Autoupdate plist | `~/Library/LaunchAgents/<label>.autoupdate.plist` | `periodic_plist_bytes` |
| Autoupdate stdout/stderr | `<root>/logs/harness-autoupdate{,.error}.log` | launchd (`StandardOutPath`/`StandardErrorPath`) |

Plist shape (the only "schema" this feature produces), as a diff against the
existing `KeepAlive` plist:

```jsonc
{
  "Label": "com.harness.autoupdate",
  "ProgramArguments": ["/bin/bash", "/root/harness-autoupdate.sh"],
  "RunAtLoad": true,
  "StartInterval": 900,              // new: replaces "KeepAlive": true
  "WorkingDirectory": "/root",
  "StandardOutPath": ".../logs/harness-autoupdate.log",
  "StandardErrorPath": ".../logs/harness-autoupdate.error.log",
  "EnvironmentVariables": { "HOME": "/Users/rem" }
}
```

CLI I/O shapes (help text / exit codes), no HTTP involved:

```
$ harness service autoupdate install --every 15m
service com.harness.autoupdate installed and started
  wrapper:  /path/to/root/harness-autoupdate.sh
  plist:    /Users/rem/Library/LaunchAgents/com.harness.autoupdate.plist
  logs:     /path/to/root/logs/harness-autoupdate.log, ...error.log
  interval: every 15m

$ harness service autoupdate install --every 90s
error: invalid --every "90s": expected <N>m, <N>h or <N>d (minutes/hours/days), e.g. "15m"
$ echo $?
2

$ harness service autoupdate status
label:    com.harness.autoupdate
plist:    ... (present)
state:    loaded
interval: every 15m

$ harness service autoupdate uninstall
service com.harness.autoupdate removed
```

## Dependencies and scope

Confirms the plan's scoping; no changes to `models.py`, `router.py`,
`dispatcher.py`, `consumer.py`, `api/`, `projection.py` тАФ invariants #1, #5
hold untouched since nothing here is a task/queue/behavior concern.

Resolved open questions from the plan (carried into the component design
above, not left open):
- Wrapper execs `harness update`, doesn't reimplement `uv tool upgrade`.
- `harness update` grows `--restart-service LABEL`; the autoupdate wrapper's
  only job is `exec` into it with the right label.
- `--every` is required тАФ no implicit default granularity.
- Thundering-herd / overlap: left to launchd's own single-instance-per-Label
  behavior, per the plan's NFR тАФ no harness-side lock file introduced.

Still open, deferred to implementation/review rather than blocking design:
- The exact fallback formatting rule in `format_interval` for a
  non-round-tripping `StartInterval` (e.g. a plist hand-edited to `5400`) тАФ
  falling back to the largest whole-minute representation (`"90m"`) is
  proposed above; flag in review if a different tie-break is wanted.
