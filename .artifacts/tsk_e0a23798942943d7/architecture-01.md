# Architecture — the `healer` workflow

Verdict up front: the plan and design are sound and I confirm them against the
current source (`app.py`, `projection.py`, `dispatcher.py`, `consumer.py`,
`task_control.py`, `models.py`, `ports/{forge,source}.py`,
`drivers/{github_client,github_forge,memory,fake_forge}.py`,
`behaviors/{landing,agent}.py`, `cli.py`, `tests/test_architecture.py`). No
port beyond `Forge.open_issue` is needed, no invariant is broken, and the one
genuinely structural change (FR-7, two workflows in one process) is a small,
targeted extension of code that was already dynamic per-task
(`Dispatcher._workflows.get(task.workflow_template)`), not a rewrite. I resolve
every "left for the architecture step" item below and give one build order.

## Alignment with existing patterns and integration points

- **`Forge.open_issue` is genuinely symmetric with `open_pull_request`.**
  `ports/forge.py` today is an ABC with one verb and one frozen result
  (`PullRequest`); adding `FiledIssue`/`open_issue` with the same
  `(task, *, ...) -> Result` shape and the same "idempotent — retry returns the
  existing one" contract is additive, not a redesign. Every implementation
  that must grow it already exists with a matching PR method to mirror:
  `GithubForge.open_pull_request` (`drivers/github_forge.py:82`),
  `MemoryForge.open_pull_request` (`drivers/memory.py:280`, idempotent by
  branch), `FakeForge.open_pull_request` (`drivers/fake_forge.py:25`,
  idempotent by branch via `prs.json`). `open_issue`'s idempotency key is
  naturally *the healer task*, not a branch — there's no branch — so each
  driver's existing bookkeeping shape (list + linear scan for `Memory`/`Fake`,
  a body marker for the real API) transfers directly.
- **`GithubClient.create_issue`/`find_issue` slot into an ABC that already
  separates "issues" from "PRs" as distinct concerns** (`list_issues` vs.
  `find_pull_request`/`create_pull_request`) and already filters PRs out of
  the issues list (`"pull_request" in item` in `HttpGithubClient.list_issues`,
  `github_client.py:170`) — the new `find_issue` reuses that exact filter.
  `FakeGithubClient` already has the `pulls`/`created` bookkeeping shape to
  mirror for `issues`.
- **`FileIssueBehavior` fits `ConsumerBehavior` with zero new obligations.**
  `LandingBehavior` (`behaviors/landing.py`) is the template: a behavior that
  touches only ports (`Forge`, here), builds its output text purely from
  `task.history` entries whose `actor` starts with `"consumer:"`
  (`_body`, `landing.py:71-78`), and returns `BehaviorResult(Outcome.DONE, …)`
  on success, letting any exception bubble to `Consumer.tick`'s existing
  `except Exception` → `_fail` path (`consumer.py:71-75`). `FileIssueBehavior`
  needs no `Workspace`/`ArtifactView` at all — simpler than landing, not an
  awkward subset of it.
- **`FailedQueueTaskSource` fits `TaskSource` exactly as designed, and dedup
  is provably correct.** I traced the full loop: `Dispatcher._fail`
  (`dispatcher.py:113-133`) and `Consumer._fail` (`consumer.py:109-127`) both
  write a `HistoryEntry(to_step=FAILED, reason=...)` — that entry is the only
  place a failure reason lives, confirming `_failure_point`'s scan
  `entry.to_step == FAILED`. `TaskControlService.restart`
  (`task_control.py:38-67`) resets `status`/`last_outcome` to `None` but
  **appends** its own history entry and keeps `history` and `id` — so
  `len(task.history)` strictly grows on every distinct failure of the same
  task id, including across a restart-then-fail-again cycle. The plan's
  dedup key, `dedup_key("failed-queue", failure.id, len(failure.history))`,
  is therefore a correct monotonic per-occurrence discriminator, not just a
  plausible one — confirmed by reading the exact code that produces both
  inputs.
- **`SourceReflectorSink` holding a live reference to `sources`, not a copy,
  is confirmed by reading its constructor** (`source_reflector.py:19`,
  `self._sources = sources`) — `build()`'s planned
  `sources.append(FailedQueueTaskSource(...))` after `failed` exists is not a
  hopeful hack, it is exactly how `SourceReflectorSink` and, later, the
  `pollers` list comprehension are already written to observe.
- **`column_order`/`BoardProjection` have exactly one call site**
  (`app.py:236`, `BoardProjection(workflow)`), so the variadic signature
  change is risk-free for every other caller.
- **`compose_prompt` needs no change**, confirmed by reading it
  (`behaviors/agent.py:83-112`): the verdict block's allowed-outcomes list is
  already built from `spec.allowed_outcomes` at runtime
  (`", ".join(outcome.value for outcome in spec.allowed_outcomes)`), so a
  `diagnose` `AgentSpec` with `allowed_outcomes=(BUG_CONFIRMED, NOT_A_BUG)`
  is enough on its own to produce the right prompt — this is the same
  mechanism this very architecture step's own prompt was generated by.
- **Guarded invariants stay intact**, confirmed against
  `tests/test_architecture.py`: `FailedQueueTaskSource` is a driver reaching
  for `ports.source`/`ports.queue`/`ports.clock`, constructed only in
  `app.py` — passes `test_only_app_and_cli_wire_drivers` and
  `test_orchestration_does_not_import_source_port` (that test only inspects
  `dispatcher.py`/`consumer.py`, both untouched). `FileIssueBehavior` reaches
  for `ports.forge`/`models` only — passes
  `test_behaviors_import_only_ports_not_drivers`. The `Outcome` extension
  passes `test_consumer_has_no_branch_on_outcome_value` for the structural
  reason the design gives (that test's AST walk forbids a comparison
  *derived from* outcome inside `consumer.py`, and adding enum members
  changes nothing there).

## Proposed architecture

Six new pieces, each a driver or behavior behind an existing port, plus one
structural change to wiring:

```
FailedQueueTaskSource ──poll()──> SourcePoller ──> inbox ──dispatcher──> diagnose queue
   (reads failed.list())                                                      │
                                                                    ClaudeCliBehavior
                                                                     (diagnose persona)
                                                                          │ bug_confirmed
                                                                          ▼
                                                                    file_issue queue
                                                                          │
                                                                  FileIssueBehavior
                                                                   .run() -> Forge.open_issue
                                                                          │
                                                                    GithubForge / MemoryForge / FakeForge
                                                                          │
                                                                       done/  (or failed/ on ForgeError)
```

The original failed task is read-only input; it is never claimed, moved, or
mutated by this feature. That is what makes healing a parallel observer
rather than a replacement for `TaskControl.restart` — both can act on the
same `failed/` task independently, and healing failing does not touch the
original task's state.

### Key decisions

**1. Dedup key: `dedup_key("failed-queue", failure.id, len(failure.history))`.**
Confirmed correct above. No alternative is warranted — a ledger inside the
driver (mirroring `GithubTaskSource._claimed`) would duplicate what
`SourcePoller._seen` already gives for free, and `GithubTaskSource._claimed`
exists to cover a *label read-after-write lag* that has no analogue here
(there is no label swap; `failed.list()` is a direct filesystem read with no
lag). Decision: no driver-local ledger.

**2. `open_issue` idempotency: a body marker via a new `find_issue`, not
Search, not a label.** Confirmed as the right call, for the reason the
design gives (Search API indexing lag vs. `list_issues`'s existing
direct-fetch pattern) plus one more: `GithubForge` already has exactly this
shape for PRs (`find_pull_request` before `create_pull_request`) — reusing
the shape means this code inherits a risk profile that is already running in
production, rather than introducing a second, different one. A label-based
approach would need a new label-management concept (create-if-missing,
naming) for no benefit over a body marker. Decision: marker
`<!-- harness-healer:<task.id> -->`, `find_issue` via
`GET /repos/{repo}/issues?state=all&per_page=100` scanned client-side,
unpaginated (matching `list_issues`'s own lack of pagination — this is not a
new, lower bar). Flag, not block: if the healer's own repo's issue count
(open+closed) exceeds 100, an older marker could be missed and a duplicate
filed. Acceptable for v1; a one-line follow-up (loop pages until `[]`) if it
ever bites — do not build it speculatively now.

**3. Multi-workflow support (FR-7) is scoped to "at most one primary +
one healer", not N workflows.** `build()` gains `heal`/`healer_repo`/
`healer_workflow_name`, computes `active_workflows = [workflow] + ([healer]
if heal else [])`, and unions `step_queues`/`BoardProjection` columns over
that list. This is deliberately not a general N-workflow scheduler (the plan
already scopes it this way) — `Harness.workflow` keeps meaning exactly what
it means today (the primary), and every other piece of `Harness`
(`step_queues`, `pollers`, `_seed_pollers`, the loops) needs zero code
change because they already operate generically over dicts/lists `build()`
hands them. Decision: adopt as designed, plus a mandatory guard — see next
point.

**4. Step-name collision: enforce at `build()` time, do not leave it a
footnote.** The plan flags this as an open question; I close it. Two
concurrently-active workflows sharing a step name is not a hypothetical —
it is exactly the kind of misconfiguration a future third workflow (or an
operator's custom `--workflow` pointed at a file that happens to reuse
`review`/`land`) would hit, and today it would silently make the second
workflow's queue steal the first's tasks with no error at all, just
mysteriously stuck or misrouted tasks days later. A four-line assertion in
`build()` (`_assert_no_step_collision`, iterating `wf.steps()` per active
workflow into an `owner: dict[str, str]`, raising on a second distinct
owner) turns that into an immediate, readable `build()`-time
`StepCollisionError`. Decision: implement it now, in the same commit as
FR-7, not as a follow-up.

**5. `harness init` always seeds `workflows/healer.json` and `agents/diagnose.json`,
independent of `--workflow`.** This is a deliberate, small behavior change to
`_init()` (today it writes exactly one workflow file, named after
`args.workflow`) — but it's the same posture the codebase already takes for
`land` (skipped from persona-seeding by name, always present in the
workflow graph) and matches "seed it, harmless unused" everywhere else in
`_init`. Decision: adopt as designed. Note for the developer: `_init()`'s
existing `build(root, args.workflow)` call must stay unchanged (it must not
pass `heal=True`) — `init` only lays down files, it never starts a loop, so
there is nothing to union there; the healer workflow file only needs to
exist on disk for `_write_default_agents` to load it via a second, direct
`FilesystemWorkflowRepository(...).get(HEALER_WORKFLOW)` call, not through
`build()`.

**6. Rollout burst (turning on `--heal` for the first time floods
one-healer-task-per-existing-failure): no time filtering, ship as designed.**
The existing `ingested` event per task already makes this visible on the
board and in the event stream — that satisfies the NFR ("a conscious,
visible event, not a silent burst") with zero new mechanism. Adding a
"since" cutoff now would be speculative scope for a v1 with no evidence yet
that backlogs are large in practice. Decision: no cutoff; document the
behavior in the CLI help text for `--heal` so an operator can choose to
clear/archive `failed/` first if they know their backlog is large.

**7. `report_progress`/`finish` on `FailedQueueTaskSource`: no-ops, as
designed.** There's no external system on the other end of a healer task
(the healer task's own origin *is* the harness's own `failed/`), so there is
nothing to project outward. Confirmed by the existing
`MemoryTaskSource`/GitHub's own no-op-when-foreign pattern via `_mine` —
this driver's no-op is the honest analogue of "nothing to do", not a stub
standing in for missing work.

## Implementation guidance

Build in this order — each step is independently testable and nothing later
depends on something earlier being wrong, so a broken step fails fast and
close to its own tests:

1. **`models.py`** — add `BUG_CONFIRMED`/`NOT_A_BUG` to `Outcome`. Trivial,
   unlocks everything else's types.
2. **`ports/forge.py`** — add `FiledIssue`, `Forge.open_issue` abstract
   method. This is a breaking-by-design change to an ABC: every existing
   concrete `Forge` (there are exactly three:
   `GithubForge`/`MemoryForge`/`FakeForge`) must implement it before
   anything instantiates cleanly again — do this and step 3 in the same
   commit/PR, or tests across the whole suite go red for the interim.
3. **`drivers/memory.py` (`MemoryForge.open_issue`), `drivers/fake_forge.py`
   (`FakeForge.open_issue` + `issues.json`), `drivers/github_client.py`
   (`IssueRef`, `GithubClient.create_issue`/`find_issue` on the ABC +
   `FakeGithubClient` + `HttpGithubClient`), `drivers/github_forge.py`
   (`GithubForge.open_issue`)** — one commit, mirrors the existing
   `open_pull_request` test suites file-for-file (a fake-client unit test,
   an HTTP-client unit test against a scripted opener, a `GithubForge` unit
   test for the marker-idempotency path and every `ForgeError` case already
   enumerated for PRs).
4. **`behaviors/file_issue.py`** — `FileIssueBehavior`, unit-tested exactly
   like `LandingBehavior`: a success path (fake forge, assert
   `BehaviorResult(DONE, ...)` and the forge recorded one issue idempotent
   on retry) and a `ForgeError` path (assert the *behavior* raises — it is
   `Consumer.tick`'s job to turn that into `failed/`, not the behavior's).
5. **`drivers/failed_queue_source.py`** — `FailedQueueTaskSource`. Unit tests:
   (a) a `default`-workflow failure yields one healer task with the right
   `dedup_key`/`data.failed_task`/`data.request`; (b) a `healer`-workflow
   failure (or any non-target workflow) yields nothing; (c) re-polling an
   unchanged `failed/` combined with `SourcePoller` yields no second task
   (this is really a `SourcePoller` behavior, so drive it through
   `SourcePoller.tick`, not by asserting on `FailedQueueTaskSource.poll()`
   alone — the dedup guarantee is `SourcePoller`'s, the source only supplies
   a correct key); (d) restart-then-fail-again (construct the second failure
   by hand with a longer `history`) yields a second, distinct task.
6. **`healer` workflow + `diagnose` persona (`cli.py`)** — `HEALER_DEFINITION`,
   `HEALER_WORKFLOW`/`FILE_ISSUE_STEP` constants, `_DIAGNOSE_PERSONA`,
   `_write_default_agents` gaining the `skip` parameter, `_init()` writing
   `healer.json` unconditionally and seeding `diagnose.json`. Test:
   `harness init` (any `--workflow`) produces `workflows/healer.json` and
   `agents/diagnose.json` but not `agents/file_issue.json`.
7. **`projection.py`** — variadic `column_order`/`BoardProjection`. Test: the
   existing single-workflow tests must pass unchanged (byte-identical
   output — this is the regression guard for "healing off changes nothing"),
   plus one new test with two workflows confirming the union and that
   `diagnose`/`file_issue` land before `done`/`failed`.
8. **`app.py`** — `build()` gains `heal`/`healer_repo`/`healer_workflow_name`,
   `_assert_no_step_collision`/`StepCollisionError`, the `behavior_for`
   branch for `FILE_ISSUE_STEP`, the `sources.append(FailedQueueTaskSource(...))`
   mutation after `failed` is constructed (put a one-line comment at that
   call site — the design is right that this is the one place a future
   reader could mistake it for dead code) and construction of
   `file_issue_behavior` unconditionally (harmless if unused, matching
   `landing`'s existing unconditional construction). Tests: `heal=False`
   builds byte-identical to today (existing `test_app.py` suite is the
   regression guard); `heal=True, healer_repo=None` raises `ValueError`;
   `heal=True` with a colliding step name raises `StepCollisionError`;
   `heal=True` with valid config produces a `Harness` whose `step_queues`
   contains both workflows' steps and whose `pollers` includes the
   `FailedQueueTaskSource`.
9. **`cli.py`** — `--heal`/`--healer-repo` on `run`, the fail-fast check
   before `build()`, passing through to `build()`, catching
   `StepCollisionError` alongside `WorkflowNotFound`.
10. **End-to-end** (`tests/test_phase4_e2e.py`-shaped): a `default` task
    fails → `FailedQueueTaskSource`/`SourcePoller` tick → healer task runs
    `diagnose` (scripted `bug_confirmed`) → `file_issue` (fake forge) →
    `done/`, one issue recorded; and the `not_a_bug` branch straight to
    `done/` with zero issues recorded. Extend `test_smoke_git.py`/
    `test_smoke_github.py` only if real-git/real-GitHub coverage surfaces a
    gap the in-memory e2e can't reach (I don't expect one: nothing here
    touches git or a worktree beyond what `ClaudeCliBehavior` already
    exercises for every other agent step).

### Contracts at a glance

```python
# ports/forge.py
@dataclass(frozen=True)
class FiledIssue:
    number: int
    url: str
    title: str

class Forge(ABC):
    @abstractmethod
    def open_issue(self, task: Task, *, title: str, body: str) -> FiledIssue: ...

# drivers/failed_queue_source.py
class FailedQueueTaskSource(TaskSource):
    kind = "failed-queue"
    def __init__(self, *, failed: TaskQueue, clock: Clock,
                 target_workflow: str = "default", healer_workflow: str = "healer",
                 healer_repo: str) -> None: ...
    def poll(self) -> list[Task]: ...             # filters failed.list() by workflow_template
    def report_progress(self, task, progress) -> None: ...  # no-op
    def finish(self, task, result) -> None: ...              # no-op

# behaviors/file_issue.py
class FileIssueBehavior(ConsumerBehavior):
    def __init__(self, *, forge: Forge) -> None: ...
    async def run(self, task: Task) -> BehaviorResult: ...   # -> forge.open_issue(...)
```

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Step-name collision between the primary workflow and `healer` silently steals tasks. | `_assert_no_step_collision` at `build()` time (decision 4) — fail loud at startup, not silently at runtime. |
| Duplicate GitHub issues from a crash between `create_issue` and the consumer recording `done`. | `find_issue` marker check before `create_issue` (decision 2) — same idempotency shape already trusted for PRs. |
| Duplicate healer tasks across restarts or repeated polls. | Delegated to the existing, already-tested `SourcePoller`/`Task.dedup_key` machinery (decision 1) — no new mechanism to get wrong. |
| A `Forge` ABC change breaks every existing caller/test at once. | Land the port change and all three implementations in one commit (build order step 2+3) — don't let the tree sit half-migrated. |
| Rollout backlog burst floods `failed/`-derived healer tasks. | Accepted per decision 6, made visible via existing `ingested` events; document in `--heal` help text. |
| The `diagnose` persona over- or under-calls `bug_confirmed`. | Not a harness-architecture concern — bounded by `allowed_outcomes` at the type level; prompt quality is a development-step/persona-tuning problem, tunable later without touching any port. |
| `find_issue`'s unpaginated `per_page=100` misses an old marker on a large healer repo. | Accepted for v1 (decision 2); documented as a one-line future fix (pagination), not built speculatively. |

## Prerequisites before implementation begins

- None outside this repository — `TaskSource`/`SourcePoller` (phase 4),
  `Forge`/`GithubForge` and `RepositoryRegistry` (phase 3) are already in
  place and confirmed above to need no shape change beyond the additions
  listed.
- Operationally (not a code prerequisite, but worth stating so it isn't
  discovered late): running `--heal` for real requires a `repos.json` entry
  for the harness's own checkout (the `--healer-repo` target) with a GitHub
  origin `GithubForge` can resolve a slug from — exactly the existing
  `open_pull_request` requirement, just pointed at this repo instead of a
  target repo.
